# PRD 생성용 메타 프롬프트 — Notion CMS 기반 개발자 포트폴리오 사이트 (MVP)

## 이 문서는 무엇인가

이 문서는 **PRD 완성본이 아니라, PRD를 작성하게 만드는 프롬프트(메타 프롬프트)** 다.
아래 "실행 프롬프트" 섹션 전체를 복사해서 Claude Code(새 대화 또는 이 프로젝트 세션)에 붙여넣으면,
Claude가 부족한 정보를 먼저 질문으로 채운 뒤 `docs/PRD.md`에 실제 PRD를 작성한다.

기능 한 줄 요약: **노션(Notion)을 CMS로 사용해, 프로젝트/이력/블로그 등의 콘텐츠를 노션에서 관리하고 웹사이트에 자동 반영되는 개발자 포트폴리오 사이트 MVP.**

## 사용법

1. 아래 "실행 프롬프트" 코드 블록 전체를 복사한다.
2. Claude Code 대화에 붙여넣고 실행한다.
3. Claude가 "0단계 — 정보 수집" 질문을 먼저 던지면 답변한다. (모르면 "MVP 기본값으로 가정해줘"라고 답해도 된다.)
4. Claude가 `docs/PRD.md`를 작성/제안한다. 승인 후 저장한다.

이 메타 프롬프트는 재사용 가능하도록 설계했다. 요구사항이 바뀌면(예: 블로그 대신 프로젝트 쇼케이스만, 노션 대신 MDX 등)
"실행 프롬프트" 안의 컨텍스트/범위 문장만 고쳐서 다시 쓰면 된다.

## 실행 프롬프트

```
당신은 노션(Notion)을 CMS로 활용하는 개발자 포트폴리오/블로그 사이트를 다수 설계해 본 시니어 프로덕트 매니저다.
지금부터 이 저장소(invoice-web)를 개발자 포트폴리오 사이트로 전환/확장하기 위한 MVP PRD를 작성한다.

## 배경 / 현재 상태
- 이 저장소는 Next.js 16 App Router + TypeScript + Tailwind v4 + shadcn/ui(base-nova, base-ui 기반) 랜딩페이지 스타터다.
- 현재 app/page.tsx는 Hero + Features + Cta로 구성된 일반 랜딩페이지이며, 포트폴리오 전용 콘텐츠(프로젝트, 경력, 블로그 등)나 노션 연동은 아직 없다.
- 컴포넌트는 L1(components/ui, shadcn CLI 생성물, 직접 수정 금지) → L2(components/layout) → L3(components/sections)로 레이어링되어 있다.
- config/site.ts가 사이트 메타/네비게이션의 단일 소스다.
- CLAUDE.md / AGENTS.md의 아키텍처 규칙과 충돌하는 제안을 하지 말 것 (예: Radix 전제, Next.js 구버전 API 가정, App Router 이전 방식 금지).

## 만들 기능 (한 줄 정의)
노션 데이터베이스/페이지에 작성한 콘텐츠(프로젝트, 소개, 경력, 기술 스택, 블로그 글 등)를 CMS처럼 사용해,
방문자가 웹사이트에서 개발자의 포트폴리오를 열람할 수 있는 MVP. 콘텐츠는 코드 수정 없이 노션에서만 갱신된다.

## 0단계 — 정보 수집 (PRD 작성 전 반드시 먼저 질문할 것)
아래 항목 중 이미 컨텍스트에서 확인 가능한 것은 스스로 판단하고, 판단 불가능한 항목만 사용자에게 질문 목록으로 물어본다.
사용자가 "MVP 기본값으로 가정해줘"라고 답하면, 각 항목에 대해 가장 단순한 MVP 기본값을 정하고 그 근거를 PRD의 "가정 및 오픈 이슈" 섹션에 명시한다.

1. 콘텐츠 범위: 노션으로 관리할 콘텐츠 타입은 무엇인가? (예: 프로젝트 목록/상세, 자기소개(About), 경력/이력, 기술 스택, 블로그 글, 연락처) — 전부 MVP에 넣을지, 일부만 넣을지.
2. 정보 구조: 노션에서 데이터베이스(예: "Projects" DB, "Posts" DB)로 관리할 항목과, 단일 페이지(예: "About")로 관리할 항목을 구분할 수 있는지.
3. 노션 연동 방식: 공식 Notion API(Integration Token) 직접 호출 vs react-notion-x 같은 렌더러 vs notion-to-md로 마크다운 변환 후 정적 생성 중 선호.
4. 데이터 갱신 전략: 빌드 타임 정적 생성(SSG, 재배포 시 반영) vs ISR(주기적 재검증) vs 요청 시 실시간 조회 중 어떤 방식을 MVP로 할지.
5. 블로그 포함 여부: 블로그(개발 글쓰기)를 MVP에 포함할지, 포트폴리오(프로젝트 쇼케이스)만으로 MVP를 한정할지.
6. 프로젝트 상세 구조: 프로젝트 하나의 상세 페이지에 필요한 필드(제목, 요약, 기간, 역할, 기술 스택, 썸네일, 링크, 본문 등)는 무엇인지.
7. 디자인/톤: 기존 config/site.ts와 랜딩 섹션(Hero/Features/Cta)을 그대로 재활용/변형할지, 포트폴리오 전용 레이아웃(예: 프로젝트 그리드, 타임라인)을 새로 설계할지.
8. SEO/공유: 프로젝트/블로그 글 상세 페이지에 OG 이미지, 메타 태그 등 SEO 요구사항이 있는지.
9. 다국어 여부: 한국어 단일 언어로 충분한지, 다국어(한/영) 지원이 MVP 범위인지.
10. 연락 수단: 방문자가 연락할 수단이 필요한지 (기존 components/sections/contact-form.tsx 재사용 여부, 이메일/소셜 링크만으로 충분한지).
11. 배포/시크릿 관리: 배포 대상(Vercel 등)과 Notion Integration Token 등 시크릿 관리 방식.
12. 도메인/브랜딩: 커스텀 도메인, 파비콘/로고 등 브랜딩 자산 준비 상태.

## 1단계 — PRD 문서 작성
0단계 답변(또는 합의된 MVP 기본값)을 반영해 아래 구조로 `docs/PRD.md`를 한국어로 작성한다.
MVP이므로 각 섹션은 "지금 만들 것"과 "지금 만들지 않을 것(Out of scope)"을 명확히 구분한다.

1. 개요 및 배경 (문제 정의: 정적 코드 수정 없이 콘텐츠를 갱신하고 싶은 니즈, 기존 포트폴리오 관리 방식의 불편함 등)
2. 목표와 성공 지표 (정성적 목표 + 측정 가능한 지표 2~3개, 예: 콘텐츠 갱신에 코드 배포 불필요, 첫 방문 로딩 속도 등)
3. 사용자 정의 (사이트 소유자(개발자 본인) / 방문자(채용 담당자, 동료 등), 각각의 니즈)
4. 핵심 유저 플로우 (소유자: 노션에 프로젝트/글 작성 → 게시 상태 변경 → 사이트에 자동 반영 / 방문자: 사이트 방문 → 프로젝트·소개 열람 → 연락)
5. 기능 요구사항 (Must-have만, 우선순위 표 형태로: 기능 / 설명 / 우선순위 P0-P2)
6. 범위 밖 (Out of scope) — 예: 댓글 기능, 조회수 집계, 관리자 대시보드, 다국어, 검색 기능 등 MVP에서 명시적으로 제외할 것
7. 노션 연동 설계 (데이터베이스/페이지 스키마, API 호출 방식, 갱신 전략(SSG/ISR/실시간), 게시 상태 필드(Draft/Published) 처리, 실패 시 폴백)
8. 정보 구조 및 라우팅 (Next.js 16 App Router 기준 라우트 설계: 홈 `/`, 프로젝트 목록/상세, 블로그 목록/상세(포함 시), About 등)
9. 화면/UX 요구사항 (각 페이지별 요구사항, 이 프로젝트의 L1/L2/L3 컴포넌트 레이어링에 맞춰 어떤 레이어에 무엇을 둘지, 기존 sections 재사용 여부)
10. SEO 및 메타데이터 요구사항 (Next.js 16 Metadata API, OG 이미지, sitemap 등 MVP 포함 여부)
11. 비기능 요구사항 (성능 — 노션 API 레이트리밋/캐싱, 접근성, 반응형, 다크모드(next-themes 기존 적용 여부))
12. 기술 스택 및 아키텍처 정합성 (Next.js 16 App Router 라우트/데이터 페칭 방식 제안, 기존 shadcn/ui 컴포넌트 및 config/site.ts 재사용 방안)
13. 마일스톤 (MVP를 더 쪼갤 수 있다면 M0/M1 단위로, 예: M0 프로젝트 쇼케이스만 → M1 블로그 추가)
14. 가정 및 오픈 이슈 (0단계에서 답을 못 받은 항목에 대해 어떤 기본값을 가정했는지 명시)

## 작성 규칙
- 전체를 한국어로 작성한다.
- MVP 원칙을 지킨다: 요청받지 않은 기능(댓글, 검색, 다국어, 관리자 대시보드 등)을 임의로 추가하지 않는다.
- 기능 요구사항은 "왜 필요한지"가 아니라 "무엇을, 어떤 조건에서" 형태로 검증 가능하게 쓴다.
- 이 프로젝트의 CLAUDE.md/AGENTS.md 아키텍처 규칙(Next.js 16 문서 우선 확인, base-ui 기반 shadcn, 컴포넌트 레이어링, config/site.ts 단일 소스)과 충돌하는 기술 제안을 하지 않는다.
- 표와 목록을 적극 활용해 훑어보기 쉽게 작성한다.
- 작성 완료 후 `docs/PRD.md` 저장 여부를 사용자에게 확인받는다 (임의로 커밋하지 않는다).
```

## 설계 메모 (참고용)

- **0단계 정보 수집을 강제한 이유**: "노션을 CMS로 쓴다"는 요구는 단순해 보이지만, 콘텐츠 범위(블로그 포함 여부)와 갱신 전략(SSG/ISR/실시간)에 따라 아키텍처가 크게 달라진다. 질문 없이 PRD를 쓰면 실제와 다른 가정이 섞여 재작업이 생긴다.
- **MVP 기본값 허용 경로를 넣은 이유**: 매번 12개 질문에 답하기 번거로울 수 있으므로, "MVP 기본값으로 가정해줘"라고만 답해도 진행되도록 탈출구를 뒀다.
- **Out of scope 섹션을 명시한 이유**: PRD 단계에서 범위가 슬금슬금 넓어지는 걸 막기 위해서다(특히 블로그/검색/댓글 등은 확장하기 쉬워 과욕을 부르기 쉽다).
- **이 프로젝트 아키텍처 규칙을 프롬프트에 직접 박아 넣은 이유**: 별도 대화/세션에서 실행해도 CLAUDE.md의 base-ui 기반 shadcn, Next.js 16 breaking changes, config/site.ts 단일 소스 같은 이 저장소 특유의 함정을 재확인하도록 하기 위함이다.
