# PRD — Notion CMS 기반 개발자 포트폴리오 사이트 (MVP)

## 1. 개요 및 배경

현재 저장소(`invoice-web`)는 Next.js 16 App Router + Tailwind v4 + shadcn/ui(base-nova) 랜딩페이지 스타터 상태다. 이를 개발자 개인 포트폴리오 사이트로 전환한다.

기존에 포트폴리오 콘텐츠(프로젝트, 블로그 글)를 관리하려면 코드에 직접 마크다운/JSX를 작성하고 매번 재배포해야 하는 번거로움이 있다. 이미 메모/기록 도구로 노션을 쓰고 있다면, 노션을 CMS로 삼아 **코드 수정 없이 노션에서 콘텐츠를 쓰고 게시 상태만 바꾸면 사이트에 반영**되도록 하는 것이 이 프로젝트의 핵심 문제 정의다.

## 2. 목표와 성공 지표

**정성적 목표**
- 프로젝트/블로그 콘텐츠를 노션에서만 작성·수정하며, 코드 배포 없이 사이트에 반영된다.
- 채용 담당자·동료가 방문했을 때 프로젝트 이력과 기술 스택을 빠르게 파악할 수 있다.

**측정 가능한 지표**
- 노션에서 콘텐츠를 게시(Published)로 변경한 뒤 최대 1시간(ISR 재검증 주기) 이내에 사이트에 반영된다.
- 신규 프로젝트/블로그 글 추가 시 코드 변경 커밋이 0건이다 (콘텐츠성 변경 기준).
- 프로젝트 상세 페이지 최초 로딩(LCP) 2.5초 이내 (일반 브로드밴드 기준).

## 3. 사용자 정의

| 사용자 | 니즈 |
|---|---|
| 사이트 소유자 (개발자 본인) | 노션에 프로젝트/글을 쓰고 게시 상태만 바꾸면 사이트에 반영되길 원함. 코드/배포 지식 없이도 콘텐츠 운영 가능해야 함 |
| 방문자 (채용 담당자, 동료, 클라이언트) | 프로젝트 이력·기술 스택을 빠르게 훑어보고, 관심 있으면 글(블로그)로 깊이를 확인하고, 연락 수단을 찾음 |

## 4. 핵심 유저 플로우

**소유자**
1. 노션 "Projects" 또는 "Posts" 데이터베이스에 새 항목을 만들고 내용을 작성한다.
2. 게시 상태(Status) 속성을 `Published`로 변경한다.
3. 별도 배포 없이, 다음 ISR 재검증 주기 내에 사이트에 자동 반영된다.

**방문자**
1. 사이트(`/`) 방문 → Hero에서 소개를 확인한다.
2. 프로젝트 목록(`/projects`)에서 하이라이트 프로젝트를 훑어보고 상세 페이지로 이동한다.
3. (선택) 블로그(`/blog`)에서 글을 읽는다.
4. 관심이 생기면 연락 섹션(Cta/Contact)을 통해 연락한다.

## 5. 기능 요구사항

| 기능 | 설명 | 우선순위 |
|---|---|---|
| 프로젝트 목록 페이지 | Notion "Projects" DB에서 `Published` 상태 항목을 카드 그리드로 표시 | P0 |
| 프로젝트 상세 페이지 | 슬러그 기반 라우팅, 본문/기술스택/기간/역할/링크 표시 | P0 |
| 블로그 목록 페이지 | Notion "Posts" DB에서 `Published` 상태 글을 목록으로 표시 (날짜 역순) | P0 |
| 블로그 상세 페이지 | 슬러그 기반 라우팅, 본문 렌더링 | P0 |
| 노션 블록 렌더러 | 문단/헤딩/이미지/코드/리스트/인용 등 기본 블록 타입 렌더링 | P0 |
| 홈 페이지 개편 | 기존 Hero를 자기소개로, Features를 대표 프로젝트 미리보기로, Cta를 연락 유도로 변형 | P0 |
| 연락 섹션 | 기존 `contact-form.tsx` 재사용 또는 이메일/소셜 링크 노출 | P1 |
| 기본 SEO 메타데이터 | 프로젝트/글 상세 페이지에 title/description 동적 생성 | P1 |
| OG 이미지 | 프로젝트 썸네일/글 커버 이미지를 OG 이미지로 사용 | P2 |
| 다크모드 | 기존 `next-themes` 적용분 유지 (추가 작업 없음) | 완료(기존 반영) |

## 6. 범위 밖 (Out of scope)

이번 MVP에서 명시적으로 제외한다.

- 별도 "About"/경력 전용 노션 DB 및 페이지 (Hero 섹션 재활용으로 대체)
- 댓글 기능
- 조회수/방문 통계 집계
- 검색 기능
- 다국어(한/영 등) 지원 — 한국어 단일
- 관리자 대시보드/노션 외 별도 CMS 백오피스
- 노션 Webhook 기반 온디맨드 재검증 (ISR 주기 재검증으로 충분, 추후 개선 과제)
- sitemap.xml / robots.txt 자동 생성

## 7. 노션 연동 설계

**데이터 소스: 노션 데이터베이스 2개**

`Projects` DB
| 속성 | 타입 | 비고 |
|---|---|---|
| 제목 | title | |
| 슬러그 | rich text | URL 경로에 사용, 고유해야 함 |
| 요약 | rich text | 카드/메타 description에 사용 |
| 기간 | rich text 또는 date range | 예: "2024.03 - 2024.06" |
| 역할 | rich text | |
| 기술 스택 | multi-select | |
| 썸네일 | files & media | 카드 및 OG 이미지 |
| 링크 | url | 배포 URL/저장소 링크 |
| 게시 상태 | status (Draft/Published) | `Published`만 노출 |
| 정렬 순서 | number (optional) | 목록 정렬용, 없으면 생성일 역순 |
| 본문 | 페이지 콘텐츠(블록) | 상세 페이지 렌더링 대상 |

`Posts` DB
| 속성 | 타입 | 비고 |
|---|---|---|
| 제목 | title | |
| 슬러그 | rich text | |
| 요약 | rich text | |
| 날짜 | date | 목록 정렬 기준 |
| 태그 | multi-select | |
| 커버 이미지 | files & media | |
| 게시 상태 | status (Draft/Published) | `Published`만 노출 |
| 본문 | 페이지 콘텐츠(블록) | |

**API 호출 방식**: 공식 Notion API(`@notionhq/client`)를 서버 컴포넌트에서 직접 호출한다. Integration Token으로 DB를 쿼리하고, 상세 페이지는 페이지 블록 children을 조회해 커스텀 렌더러로 변환한다.

**커스텀 블록 렌더러 (MVP 범위)**: paragraph, heading_1~3, image, code, bulleted/numbered list, quote만 지원한다. 그 외 블록 타입(테이블, 임베드, 콜아웃 등)은 MVP에서 지원하지 않으며 텍스트로 폴백하거나 무시한다.

**갱신 전략**: ISR. 각 라우트에 재검증 주기(예: `revalidate = 3600`, 1시간)를 설정해 재배포 없이 주기적으로 최신 데이터를 반영한다. 실시간성이 중요하지 않은 개인 포트폴리오 특성상 온디맨드 재검증은 MVP에서 제외한다.

**게시 상태 처리**: `Draft` 상태 항목은 목록/상세 어디에도 노출하지 않는다. 상세 페이지 접근 시 슬러그가 `Draft`거나 존재하지 않으면 404 처리한다.

**실패 시 폴백**: 노션 API 호출 실패 시 빌드/재검증 시점의 캐시된 마지막 성공 데이터를 유지한다(ISR 특성상 자동 보장). 완전 초기 빌드 실패 시에는 빈 목록 대신 에러 상태를 표시한다.

## 8. 정보 구조 및 라우팅

Next.js 16 App Router 기준:

```
/                  홈 (Hero + 대표 프로젝트 미리보기 + Cta/연락)
/projects          프로젝트 목록
/projects/[slug]   프로젝트 상세
/blog              블로그 목록
/blog/[slug]       블로그 상세
```

`config/site.ts`의 `navItems`에 "프로젝트", "블로그" 항목을 추가하고, `siteConfig.name`/`description`을 포트폴리오 성격에 맞게 수정한다 (단일 소스 원칙 유지).

## 9. 화면/UX 요구사항

기존 컴포넌트 레이어링(L1 `components/ui` → L2 `components/layout` → L3 `components/sections`)을 그대로 따른다.

- **홈(`/`)**: 기존 `Hero`/`Features`/`Cta` 섹션을 재사용하되 문구·데이터를 포트폴리오 성격으로 변경한다. `Hero`는 자기소개, `Features`는 대표 프로젝트 3~4개 미리보기, `Cta`는 연락 유도 역할로 변형한다. 새 레이아웃을 처음부터 설계하지 않는다.
- **프로젝트 목록/블로그 목록**: L3 신규 섹션(`components/sections/project-list.tsx`, `components/sections/post-list.tsx` 등)으로 추가하고, 카드 UI는 기존 L1 `card.tsx`, `badge.tsx`(기술 스택 태그)를 재사용한다.
- **상세 페이지**: 노션 블록 렌더러 출력을 감싸는 타이포그래피 스타일을 Tailwind로 정의한다.
- **연락**: 기존 `components/sections/contact-form.tsx`를 재사용한다.
- 반응형: 기존 스타터의 반응형 기준(모바일 nav 등)을 그대로 따른다.

## 10. SEO 및 메타데이터 요구사항

- Next.js 16 Metadata API(`generateMetadata`)를 프로젝트/블로그 상세 페이지에 적용해 노션의 제목/요약을 title/description으로 사용한다.
- OG 이미지는 노션의 썸네일/커버 이미지를 그대로 사용한다 (별도 OG 이미지 생성 파이프라인은 MVP 제외).
- sitemap.xml, robots.txt 자동 생성은 P2 이후 과제로 보류한다.

## 11. 비기능 요구사항

- **성능/레이트리밋**: 노션 API 호출은 ISR 캐시로 감싸 요청마다 직접 호출되지 않도록 한다. 목록 페이지는 페이지네이션 없이 전체 조회하되, 항목 수가 많아지면 추후 페이지네이션을 검토한다(오픈 이슈).
- **접근성**: 기존 shadcn/ui(base-ui) 컴포넌트의 접근성 특성을 유지하고, 노션 블록 렌더러의 이미지에는 alt 텍스트(노션 캡션 활용)를 넣는다.
- **반응형**: 기존 스타터의 브레이크포인트 기준을 따른다.
- **다크모드**: 기존 `next-themes` 적용을 그대로 유지하며 추가 작업은 없다.

## 12. 기술 스택 및 아키텍처 정합성

- Next.js 16 App Router의 서버 컴포넌트에서 `@notionhq/client`로 데이터를 페칭한다. Route Segment Config(`revalidate`)로 ISR을 구성한다 — 구체적인 API 시그니처는 `node_modules/next/dist/docs/`를 확인해 이 버전 기준으로 구현한다 (AGENTS.md 지침).
- 신규 의존성: `@notionhq/client` (공식 SDK) 1개만 추가한다. react-notion-x, notion-to-md 등은 채택하지 않는다.
- 기존 shadcn/ui(base-ui 기반) 프리미티브(`card`, `badge`, `skeleton` 등)를 재사용하고, `components/ui/*`는 직접 수정하지 않는다.
- `config/site.ts`를 내비게이션/메타의 단일 소스로 계속 사용한다.
- 환경변수: `NOTION_TOKEN`, `NOTION_PROJECTS_DATABASE_ID`, `NOTION_POSTS_DATABASE_ID`를 `.env.local`(로컬) 및 배포 플랫폼의 환경변수(예: Vercel)로 관리하고 저장소에 커밋하지 않는다.

## 13. 마일스톤

| 단계 | 범위 |
|---|---|
| M0 | 프로젝트 쇼케이스: 노션 연동, 프로젝트 목록/상세, 홈 개편 |
| M1 | 블로그: 블로그 목록/상세, 노션 블록 렌더러 확장 |

## 14. 가정 및 오픈 이슈

- **배포 대상**: 별도 확인 없이 Vercel을 기본 가정한다 (Next.js App Router + ISR과 궁합이 좋음). 다른 플랫폼 사용 시 ISR 지원 여부를 재확인해야 한다.
- **도메인/브랜딩**: 커스텀 도메인, 로고/파비콘 준비 여부는 확인되지 않아 오픈 이슈로 남긴다. 준비되는 대로 `config/site.ts`와 `app/favicon.ico`를 갱신한다.
- **프로젝트 상세 필드**: 제목/슬러그/요약/기간/역할/기술스택/썸네일/링크/본문을 기본 스키마로 가정했다. 실제 운영하며 필드가 추가/축소될 수 있다.
- **다국어**: 한국어 단일로 가정했다. 추후 다국어가 필요하면 별도 PRD로 분리한다.
- **About 페이지 미포함**: 콘텐츠 범위 결정 시 별도 About 페이지 대신 홈의 Hero 섹션이 자기소개 역할을 겸하는 것으로 가정했다. 운영해보고 부족하면 M2 이후 별도 페이지로 분리한다.
